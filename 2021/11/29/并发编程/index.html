一、基本概念

进程线程

粗略理解： 一个程序可以有好几个进程，一个进程又可以有好几个线程。线程(thread)是计算机调度资源的最小单位。

任务管理器中出出现程序的进程，举例 idea 软件 可以有3，4个进程，而每一个进程下面有可以有好多进程。

  • 一个进程可以并发执行多个线程 （多线程并发）
  • 程序 –> 进程 —> 线程 （ 从上到下，下级单位）
  • 每个线程是独立的

线程的状态

wait 与 sleep 方法

区别

  • wait方法是 Object类的方法，因为Object类是所有类的父类，所以所有实例对象都可以调用 wait方法，而sleep方法是Thread类的静态方法。

  • 最主要是sleep方法没有释放锁，而wait方法释放了锁，使得其他线程可以使用同步控制块或者方法。

  • sleep 让线程从 【running】 -> 【阻塞态】 时间结束/interrupt -> 【runnable】

  • wait 让线程从 【running】 -> 【等待队列】 notify/notifyAll-> 【锁池】 -> 【runnable】

并发与并行

  • 并发 多个线程去访问同一个资源 例子：春运抢票 商城秒杀
  • 并行 多个线程同时进行工作。

管程

  • 管程 —- monitor 监视器（操作系统层面） —–java中的锁（java层面）

  • 是一种同步机制 保证同一时间只有一个线程访问资源

  • jvm 中同步是基于管程实现的，进入临界区加锁，退出临界区释放锁

用户线程和守护线程

  • 用户线程：用户自定义线程
  • 守护线程：（后台线程） 比如 垃圾回收线程
  • 主线程结束了，用户线程还没结束 ，jvm存活
  • 没有用户线程了，守护线程也就结束，jvm 结束
  • 守护线程是依赖于用户线程，用户线程退出了，守护线程也就会退出，典型的守护线程如垃圾回收线程。
  • 用户线程是独立存在的，不会因为其他用户线程退出而退出。

默认情况下启动的线程是用户线程,通过Thread类的setDaemon(true)将线程设置成守护线程,这个函数务必在线程启动前进行调用，否则会报java.lang.IllegalThreadStateException异常,启动的线程无法变成守护线程,而是用户线程。

二、Lock接口

synchronized关键字

synchronized是一种同步锁，有以下三种用法

  • 修饰 方法，称为同步方法，作用范围就是这个方法，作用对象就是调用这个方法的对象

  • 同步代码块，作用范围就是{}中的代码，作用对象即使调用这个代码块的对象

  • 修饰静态方法，作用对象就是这个类的所有对象

  • 修饰一个类

什么叫可重入锁

  • synchronized与ReentrantLock都是可重入锁

可重入锁，也叫做递归锁，指的是同一线程外层函数获得锁之后 ，内层递归函数仍然可以获取该锁的代码，但不受影响

可重入就是说某个线程已经获得某个锁，可以再次获取锁而不会出现死锁。

可以多次获取同一个锁，但是也需要挨个去解锁，即是获取同一个锁五次那么也需要解锁五次

//已经获取锁的情况下，可重复多次获取这个锁
public class ReenLock {

    public void sdf() {
        new Thread(()->{
            synchronized (this) {
                System.out.println("第1次获取锁，这个锁是：" + this);
                int index = 1;
                while (true) {
                    synchronized (this) {
                        System.out.println("第" + (++index) + "次获取锁，这个锁是：" + this);
                    }
                    if (index == 10) {
                        break;
                    }
                }
            }
        },"AA").start();
    }

    public static void main(String[] args) {
        new ReenLock().sdf();
    }
}

Lock和 synchronized 区别

  • synchronized 自动上锁解锁，lock 手动上锁解锁
  • synchronized 放生异常时自动释放锁，lock发生异常时不会释放锁，所以 lock.unlock()写在finally中，防止发生异常不释放锁
  • lock 用途更加多，ReentrantLock提供了其他的功能，包括定时的锁等待、可中断的锁等待、公平与非公平锁。而sychronized没有这些功能
public class TicketReenLock {
    private int ticketNum = 100;
    //声明 lock 对象  声明可重入锁
    private final Lock lock = new ReentrantLock();

    public  void sale() {
        //获取锁
        lock.lock();
        try {

            if (ticketNum > 0) {
                System.out.println(Thread.currentThread().getName()+"卖出"+(ticketNum--)+"号票"+"还剩下"+ticketNum+"张票");
            }

        } catch (Exception e) {
            e.printStackTrace();
        }finally {
            //释放锁
            lock.unlock();
        }

    }

    public static void main(String[] args) {

        TicketReenLock lock = new TicketReenLock();
        //函数式接口编程
        new Thread(()->{
            System.out.println("11");
            for (int i = 0; i < 100; i++) {
                lock.sale();
            }
        },"BB").start();

        new Thread(()->{
            for (int i = 0; i < 100; i++) {
                lock.sale();
            }
        },"cc").start();
    }

}

解析 虚假唤醒为什么出现

  • wait()方法会释放锁，其他线程会获取锁

  • wait()方法会让线程进入到等待状态，waiting ，在哪里等待就会在哪里重新接着执行

  • if方法中的wait()方法中被唤醒后，并不会再去判断 num的值，看是否在需要去执行wait()方法

  • 避免虚假唤醒这种情况出现，必须使用while去判断num的值，来决定是否去执行wait()方法，当wait()方法被 notify()/notifyall()方法唤醒后，会继续去判断是否 满足 条件

//加1 方法
public synchronized void incr() throws InterruptedException {
	
    //虚假唤醒
    if (num != 0) {
        //当线程从wait（）方法中被唤醒后，不会再去判断 num的值，即是if判断只会在第一次会执行，当线程后等待状态被唤醒后，不会再去执行if判断
        wait();
    }
    num++;

    System.out.println(Thread.currentThread().getName()+":"+num);

    //通知
    notify();
}

//减一1 方法
public synchronized void desc() throws InterruptedException {
    
    //使用while 去避免 虚假唤醒
   while(num !=1) {
        wait();
    }

    num--;

    System.out.println(Thread.currentThread().getName()+":"+num);

    notify();
}
